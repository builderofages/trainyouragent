This is the playbook. Lift it.

## The thesis

Most B2B SaaS sites have 8-12 pages: home, pricing, features, about, blog, contact. They all target high-volume head terms ("ai voice agent," "ai customer support") where they're competing against companies that raised $40M.

The unfair advantage for a small operator is the **long tail**. Nobody is writing landing pages for "ai voice agent for plumbing companies in cleveland." But that's exactly what the small Cleveland plumbing owner is googling. If you show up there, the click-through-to-book rate is ~10x what you'd get from a generic search.

The math: 200 pages × even a tiny 30 monthly impressions × even a 2% click rate × even a 5% conversion = ~6 leads/month. Stack on enough verticals × cities and you have a meaningful channel that costs you zero in ad spend.

## The template (one component, two arrays)

We had a vertical × city template. 14 verticals × 15 cities = 210 pages. Here's the entire data layer:

```ts
// src/content/locations.ts
export const VERTICALS = [
  { slug: "hvac", label: "HVAC", problem: "after-hours emergency calls" },
  { slug: "healthcare", label: "Healthcare", problem: "patient intake + insurance verification" },
  { slug: "real-estate", label: "Real Estate", problem: "speed-to-lead on Zillow inbound" },
  { slug: "legal", label: "Legal", problem: "intake + conflict-check" },
  // ... 10 more
];

export const CITIES = [
  { slug: "tampa", label: "Tampa", state: "FL", metro: "Tampa Bay" },
  { slug: "miami", label: "Miami", state: "FL", metro: "Miami-Dade" },
  // ... 13 more
];
```

And the route:

```tsx
// src/App.tsx
<Route path="/:vertical/:city" element={<LocationPage />} />
```

And the page component (skeleton):

```tsx
function LocationPage() {
  const { vertical, city } = useParams();
  const v = VERTICALS.find(x => x.slug === vertical);
  const c = CITIES.find(x => x.slug === city);
  if (!v || !c) return <NotFound />;

  return (
    <>
      <Helmet>
        <title>AI Voice Agent for {v.label} in {c.label}, {c.state} — TrainYourAgent</title>
        <meta name="description" content={`24/7 AI voice agent for ${v.label} companies in ${c.label}. Built for ${v.problem}. 14-day setup.`} />
        <link rel="canonical" href={`https://trainyouragent.com/${v.slug}/${c.slug}`} />
      </Helmet>
      <Hero vertical={v} city={c} />
      <Problem vertical={v} city={c} />
      <SocialProof vertical={v} />
      <CTA vertical={v} city={c} />
    </>
  );
}
```

That's the whole architecture. ~120 lines of code. Vite + React + react-router-dom. No CMS, no database, no rebuild pipeline beyond `git push`.

## The content layer

The page that ranks is not the page with the most words. It's the page that **matches the intent of the search** more precisely than the competitors. For our long-tail combos, the searcher intent is roughly:

- I run a [vertical] business in [city]
- I'm wondering if AI can help with [vertical's biggest pain]
- I want to know what it costs and how fast we can get started

So the page hits each of those in order: a city-specific hero, a vertical-specific problem statement, social proof in the same vertical, pricing, and a calendar booking widget. ~600 words per page. We rotate three variants of each section so Google doesn't see 200 identical pages with just the city name swapped (that's what gets you a doorway-pages penalty).

## The sitemap regen

The single most-skipped step in programmatic SEO is the sitemap. Google won't crawl pages it doesn't know about, and 200 long-tail pages buried under a SPA router are invisible until they're in sitemap.xml.

Our sitemap generator runs at build time:

```ts
// scripts/generate-sitemap.ts
import { VERTICALS, CITIES } from "../src/content/locations";
import fs from "fs";

const urls = [];
for (const v of VERTICALS) {
  for (const c of CITIES) {
    urls.push(`https://trainyouragent.com/${v.slug}/${c.slug}`);
  }
}

const xml = `<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
${urls.map(u => `  <url><loc>${u}</loc><changefreq>monthly</changefreq><priority>0.7</priority></url>`).join("\n")}
</urlset>`;

fs.writeFileSync("public/sitemap.xml", xml);
```

Wire it into `package.json`:

```json
"scripts": { "build": "tsx scripts/generate-sitemap.ts && vite build" }
```

Sitemap regens every deploy. Submit it once in Search Console. Done.

## The three things we screwed up

This wasn't clean. Three things broke and what fixed them:

**1. We didn't 301 the unused combos.** Some vertical × city combos make no sense ("ai voice agent for ski lodges in miami"). We generated all 210 then realized ~20 were nonsense. We didn't take them down, we just left them — and Google flagged them as thin content. Fix: delete or 301 the unused combos to the parent vertical page.

**2. We launched without internal links.** First two weeks the pages were orphaned. Google found them via sitemap but they had no PageRank flowing in. We added a "Cities" footer block on each vertical page linking to all city variants. Rankings climbed within 14 days.

**3. We forgot the FAQPage schema.** First-version pages had Article schema only. Adding FAQPage schema (5 Q&As per page) earned rich-snippet placements on ~12% of queries — those have ~3x CTR. We're now adding FAQPage schema to every vertical and pillar page (this PR, in fact).

## When this doesn't work

Programmatic SEO breaks when:
- Your verticals/cities aren't actually distinct buyers (you're just duplicating content)
- You're in a market where the head terms aren't dominated (just go after the head)
- You can't fulfill — if the page books a call but you can't service Tampa HVAC, you're burning trust

If you can fulfill, you have ≥5 distinct verticals, and you have ≥10 markets, programmatic SEO will probably print money for you. Lift the template.
